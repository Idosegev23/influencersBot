# Agent-WhatsApp Blueprint — Implementation Plans (index)

4 per-phase TDD plans (P0-P3) + lead-architect review. Spec: `docs/superpowers/specs/2026-07-08-agent-wa-blueprint.md`.
Scope overrides honored: NO buttons/list-picker · free-form voice · read-back confirmation · model+context entity resolution.

## Build order

1. P0 (money correctness) FIRST — self-contained on the existing synchronous engine; it establishes the four seams every later phase consumes: the {reply,outcome} contract on handleAgentMessage (today it returns Promise<string|null> — verified), normalizeAmount() in wa-interpret, issueQuote() idempotency 3rd arg (today it is 2-arg — verified), and the crm_agent_wa_log table. Correct as the leadoff.
2. P1 (reliability: QStash worker + per-agent lock + state TTL + free-form read-back + fuzzy resolver) SECOND — hard-depends on P0's {reply,outcome} (runAgentJob picks reaction from outcome) and on normalizeAmount (the read-back must echo the normalized amount). Moves the exact handleAgentMessage call into the worker. Correct.
3. P2 (STT + dual-provider model layer) THIRD — depends on P0's normalizeAmount seam and on P1's worker (Gemini-Pro native audio exceeds the 300ms webhook budget so must run in the worker) and P1's free-form confirm stage (voice read-back reuses it). Correct.
4. P3 (RAG brain + proactivity + eval) LAST — depends on P0 (normalizeAmount + log rows seed golden cases), P1 (worker runs auto-ingest under the lock; fuzzy resolver), and P2 (llm layer + router + transcribeHebrew). Correct as the capstone.
5. NO reordering needed: the declared dependency graph is a clean linear chain P0→P1→P2→P3 with no cycles. The only build-order hazard is migration numbering (see dependency_issues), not phase sequence.

## Dependency issues
- MIGRATION-NUMBER COLLISION (three-way): P0 produces 061_wa_idempotency_and_log.sql, P1 produces 061_wa_agent_p1.sql, and P3 produces 061_crm_agent_embeddings.sql + 062_crm_agent_wa_memory.sql + 063_crm_agent_nudges.sql — three phases all claim 061 (P3 also reuses 062/063 which will exist by then). Highest existing migration is 060 (verified). For a coherent P0→P3 build these must be sequential: P0=061, P1=062, P3=063/064/065. The repo already tolerates duplicate numbers (two 053–059 each) via apply_migration, but P0, P1 and P3 all naming 061 will overwrite/conflict. P2 sidesteps this by using a timestamp form (20260708_agent_wa_stt_audit.sql) — inconsistent convention but no numeric clash.
- UNOWNED ROUTING SEAM (money vs advisory lane): P2 produces routeIntent() returning {lane:'money'|'advisory',...} but P2's wa-conversation changes only rewire aiPricing/planFreeform through extractStructured — nothing in P2 actually calls routeIntent to branch the worker. P3's own open_questions flag this ('does P2's Stage-A router own the lane branch, or does P3 add it?'). No phase definitively owns wiring the router into handleAgentMessage/the worker as the money-vs-advisory dispatch point. This is the central architectural seam of the spec (§1 two lanes) and it falls between P2 and P3.
- ORPHANED ANOMALY-HISTORY GATE: P0 ships AMOUNT_ANOMALY_FACTOR=6 and the reason='anomalous' path but leaves opts.history UNWIRED, explicitly deferring the talent/brand-history query 'to P1'. P1's produces list contains no history lookup or anomaly wiring (it is async/lock/TTL/resolver only), and P2/P3 do not pick it up either. Spec §5.4's '>X% deviation from talent/brand history → needs_confirmation' therefore has no owning task — the code path ships dormant forever.
- STT NOT THREADED THROUGH P1's WORKER CONTRACT: P1 defines AgentJob {waId,agentId,msg,textBody} and runAgentJob calls handleAgentMessage, but neither carries an stt field. P2 requires handleAgentMessage opts extended to {isVoice?; stt?:{confidence,provider}} and needs runAgentJob/materializeInbound to pass it. Sequentially fine, but P2 must retrofit P1's runAgentJob + AgentJob shape; neither plan states P2 edits runAgentJob's signature, so the seam is implicit.
- FILE MISATTRIBUTION (minor): P0 lists 'src/lib/crm/quotes.ts :: issueQuoteForDeal is called with a deterministic key' under the quotes.ts heading, but issueQuoteForDeal actually lives in src/lib/crm/wa-conversation.ts (verified, line 424), not quotes.ts. P0's consumes list correctly places it under wa-conversation, so this is just a mislabeled produce line, but it could mislead the implementer about which file to edit.

## Interface mismatches
- reactionForOutcome reaction set: P0 defines reactionForOutcome(o:AgentOutcome):'✅'|'⚠️'|null (outcomes done|need_more|error). But P1's runAgentJob description says it chooses '✅ vs ⚠️ vs 👍' — 👍 is not a value reactionForOutcome can return, and there is no fourth outcome to map it from. The need_more→⚠️ vs need_more→👍 mapping is ambiguous; P1 should consume reactionForOutcome verbatim rather than introduce a 👍 the P0 contract cannot produce. (Spec uses 👀 for ack and ✅ for done only.)
- Raw vs normalized amount in the voice read-back: P2's produces defines criticalAmounts(text) via wa-interpret.extractNumbers() (RAW numbers), yet P2's own dependency note says confidence.ts 'relies on normalizeAmount ... without it the read-back echo and the LLM pricing output disagree on 80 vs 80,000'. Using extractNumbers (unscaled) for the STT critical-number echo while the money path uses normalizeAmount (scaled) reintroduces exactly the 80-vs-80,000 divergence the spec §5.4 says the single normalizer must kill. The critical-number gate must run through normalizeAmount, not extractNumbers.
- handleAgentMessage opts drift across phases: current sig opts={isVoice?} (verified). P0 changes only the RETURN type to AgentMessageResult; P2 extends opts to {isVoice?; stt?:{confidence,provider}}. P1 and P3 both reference handleAgentMessage(agent,waId,text,attachments,opts) generically without the stt field. No hard conflict, but the opts shape is defined in three places and only P2 pins the stt sub-shape — the implementer must treat P2's opts as the canonical superset.

## SCOPE VIOLATIONS (buttons/picker sneaking back)
- P0 keyword-only confirmation persists: P0's consumes list keeps interpretYesNo() and P0 threads normalizeAmount().needsConfirmation into 'the read-back', but the confirmation REPLY in P0 is still interpreted by interpretYesNo (deterministic keyword set) — i.e. P0 ships read-back-then-keyword-confirm, not read-back + free-form + model-in-context. Per the SCOPE ADJUSTMENTS this is the brittle keyword confirmation the spec wants gone. Acceptable ONLY as an interim state because P1 explicitly replaces it with interpretConfirmReply (deterministic fast path + model-in-context fallback); flagging per instruction to flag every instance. If P1 slips, P0 alone violates the free-form-confirm mandate.
- NO buttons / list-picker reintroduced in any of the four plans — verified clean. Note for clarity: P1's classifyConfirm(text):'yes'|'no'|'other' and P0/P1 extractNumbers/interpretYesNo are deterministic fast paths, NOT interactive WhatsApp buttons or list messages, and P1 pairs classifyConfirm with a model-in-context fallback (interpretConfirmReply) — so P1/P2/P3 are compliant with the read-back + free-form + context rule. No interactive-message API is proposed anywhere.

## Gaps
- DRAFT EMAIL / COUNTER-OFFER capability is unowned. Spec North Star §0 ('draft a reply to the brand'), §2C ReAct tool draft_message, §4 matrix #7 (draft emails/counter-offer → GPT-5.5), and §4.5 all require it, but P3's agent-tools produces only countContracts/listContracts/sumSales/getQuoteDetails/talentStats/pipelineStatus/revenueByPeriod/searchContext — no draft_message / message-drafting tool. A headline advisory-lane feature has no task.
- MODEL MATRIX #6 (brief PDF/image parse → Gemini 3 Pro vision) is unowned. P2's llm/config.ts Lane type is only 'money'|'router'|'stt'|'money_fallback' — no vision/brief-parse lane. The attachment brief-intake path (startBrief → ai-parser) is never re-modeled by any phase, so the spec's multimodal brief-parse upgrade is dropped.
- STORE-THE-AUDIO requirement (§3.4 'Store the audio (or media_id + transcript) for debugging + eval') is only half owned. P2 stores stt_confidence/stt_provider on crm_inbound_messages and transcript in crm_agent_wa_log, but no task persists the raw audio bytes or media_id for later replay/eval — the STT golden/eval set in P3 has no audio fixtures to replay against.
- KPI COMPUTATION (§6.2) has no owning task beyond raw logging. crm_agent_wa_log captures the columns, and P3 ships an analytics dashboard over agent-tools (revenue/counts), but nothing computes or surfaces the operational KPIs the spec names: pricing-correction-rate, router-accuracy-vs-golden, STT-confidence distribution + %read-backs, model-fallback-rate, cost/conversation, voice-resend-rate.
- STT CONFIDENCE CALIBRATION eval (§7 'Confidence calibration — when the model is sure, how often is it right') is unowned. P3's eval (scoreMoney/scoreLeak + LLM judge) covers money exactness and leakage on text scenarios, but there is no task that calibrates STT self-reported confidence against ground truth — which is also what the P2 read-back threshold (0.75) needs to be tuned against.
